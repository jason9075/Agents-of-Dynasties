# Agents of Dynasties — TODO

這是為了修補現有機制與處理邊緣狀況（Edge Cases）的待辦事項清單：

## 1. 戰爭迷霧與索敵（Fog of War & Targeting）漏洞
- **目標追蹤機制調整**：當使用 `ATTACK` 鎖定的目標進入戰爭迷霧而失去視野時，單位不會具備全域追蹤能力。單位會移動到「最後一次看見目標的格子」，如果到達該格後仍未發現目標，則指令失效。

## 2. 資源、人口的「取消與退還」機制缺陷 done
- **新增 `CANCEL_PRODUCE` 動作**：允許玩家取消建築中的生產佇列。
- **全額退款機制**：當玩家主動取消生產佇列，或是包含生產佇列的建築遭到摧毀時，系統必須「全額」退還預扣的資源與已保留的人口數。
- **新增無代價自我消滅（Delete）功能**：新增允許玩家主動刪除/處決己方單位的 API，刪除單位不需要付出任何資源代價，以解決遊戲後期 20 人口上限造成的兵種轉型死鎖問題。

## 3. 時序結算（Tick Resolution）的連鎖失效
- **卸貨失敗處理**：當村民採集完畢準備卸下資源，但目標 `town_center` 在同一個 Tick 內被提早摧毀時，村民會因為找不到卸貨目的地，而把資源繼續放在身上並轉入 `IDLE` 狀態。

## 4. 實體建築與空間阻擋策略
- **限制單人建造**：限制每個建築只能由一位村民進行建造，不允許多人協同建造。
- **死胡同尋路處理**：如果地圖被完全建築物堵死，導致尋路失敗時，單位不會主動破牆，而是直接轉入 `IDLE` 狀態。

## 5. 終局判定（Win Conditions & Draw）done
- **勝利條件**：摧毀敵方城鎮中心（Town Center）。
- **平手判定**：如果在同一個 Tick 內，雙方互相將對方的城鎮中心拆除（血量同時歸零），這場比賽將會判定為平手（Draw）。

## 6. 戰鬥與單位的 AI 本能（Tactical Behavior）done
- **發呆受擊反擊**：如果單位處於 `IDLE` 狀態遭到攻擊，會自動進行反擊。但如果是正在採集資源的村民（`GATHER`），則會無視攻擊，繼續執行收集資源的動作。
- **地形阻擋視野（LOS Blocking）**：高山（`mountain`）和森林（`forest`）會造成實際的視野遮蔽，單位無法看透這些地形。

## 7. 建築耐久與防禦機制（Building & Survival Mechanics）
- **無維修與進駐機制**：目前維持「無」此類機制的設定。建築物受到傷害後不可由村民維修，村民與單位也無法進駐城鎮中心（Town Center）或防禦建築躲避攻擊。

## 8. API 資訊洩漏與戰爭情報（Information Security in API）
- **生產情報公開（作為遊戲特色）**：當玩家視野看見敵方建築時，API 可以取得該建築物「正在生產什麼單位」。這被視為一種供 AI Agent 利用的優勢（例如：指派斥侯偵查以獲得科技情報）。

## 9. 決策衝突與防呆（Command Resolution Conflicts）
- **API 衝突阻斷**：當對多個村民/單位下達會互相衝突的指令（例如試圖讓多名村民蓋同一棟建築）時，多餘的請求（Request）會直接被 API 拒絕（Reject），並且在回應中附上具體的失敗原因。

## 10. 目標優先級（Targeting Priority）
- **維持原始設定，交由代理管理**：系統內建的索敵邏輯不變（最近距離、最低血量）。若單位發起無效率的攻擊（例如無意義地攻擊閒置建築），這屬於 AI Agent 必須自行發現並手動微操管理的責任範圍。

## 11. 資源爭奪（Resource Contention）時序規則
- **依賴 ID 順序結算**：在同一個 Tick 內若發生多餘單位的資源爭奪（例如：城鎮中心生產與村民蓋建築同時競爭剩餘的木頭），將直接維持使用「單位 ID 順序」作為依序結算的標準。

## 12. 出怪格決定論與集結點（Spawn Determinism & Rally Point）done
- **預設固定生成順序**：單位生產完畢後，預設一律從建築的「右下方」為首選生成點。如果該格子有實體阻擋，則往周圍尋找下一個合法的放置點。
- **新增集結點設定 API**：在建築物的 API 動作中新增設定，允許玩家/Agent 設定該建築物的「出生優先格數位置（Rally Point）」，以掌控兵種的降落方位。
